## рџЏ›пёЏ Architecture

This project follows **Clean Architecture** organized **feature-first**. Every feature is an isolated module; the three architectural layers live inside each feature rather than spanning the whole project.

```
Feature
в”њв”Ђв”Ђ data/          в†ђ External world: Firebase, models, repository impl
в”њв”Ђв”Ђ domain/        в†ђ Pure Dart: entities, repository contracts, use cases
в””в”Ђв”Ђ presentation/  в†ђ Flutter: BLoC, pages, widgets
```

### The Three Layers

**Data Layer**
Responsible for communicating with external data sources (Firebase Auth, Firestore). Contains:
- `datasources/` вЂ” Remote data source classes that wrap Firebase SDK calls
- `models/` вЂ” Data transfer objects (DTOs) annotated with `@freezed`, serializable to/from Firestore maps. Generated via `build_runner`.
- `repositories/` вЂ” Concrete implementations of domain repository interfaces, converting models в†’ entities and wrapping errors in `Either<Failure, T>` via `dartz`.

**Domain Layer**
The heart of the app вЂ” pure Dart, zero Flutter or Firebase dependencies. Contains:
- `entities/` вЂ” Plain Dart classes representing business objects (e.g. `UserEntity`, `MessageEntity`)
- `repositories/` вЂ” Abstract interfaces (contracts) that the data layer must fulfil
- `usecases/` вЂ” Single-responsibility classes, each encapsulating one business action (e.g. `SendMessageUseCase`, `LoginUseCase`). Called directly from BLoC.

**Presentation Layer**
Flutter-specific UI logic. Contains:
- `bloc/` вЂ” BLoC classes using `flutter_bloc`. Events and states are `@freezed` sealed unions for exhaustive pattern matching.
- `pages/` вЂ” Full-screen route-level widgets
- `widgets/` вЂ” Smaller, reusable UI components scoped to the feature

### Data Flow

```
UI Widget
  в†’ dispatches BLoC Event
    в†’ BLoC calls UseCase
      в†’ UseCase calls Repository (abstract)
        в†’ Repository Impl calls DataSource
          в†’ DataSource queries Firebase
        в†ђ Returns Either<Failure, Model>
      в†ђ Maps to Entity
    в†ђ BLoC emits new State
  в†ђ Widget rebuilds via BlocBuilder
```

### Core Layer

Shared infrastructure used by all features lives in `lib/core/`:

| Folder | Purpose |
|---|---|
| `di/` | `injectable` module вЂ” registers all dependencies into the `get_it` service locator |
| `error/` | `Failure` sealed classes and `Exception` types |
| `router/` | `go_router` route definitions and guards |
| `theme/` | App-wide theme, colors, and `google_fonts` typography |
| `utils/` | Shared constants and helper utilities |

## рџљЂ Getting Started

### Prerequisites

- Flutter SDK `^3.12.2` or later
- Dart SDK `^3.12.2` or later
- A Firebase project with **Authentication** and **Cloud Firestore** enabled
- `flutterfire_cli` installed globally

### 1. Clone the repository

```bash
git clone https://github.com/2001arman/flutter_clean_architecture_bloc.git
cd flutter_clean_architecture_bloc
```

### 2. Install dependencies

```bash
flutter pub get
```

### 3. Run code generation

This project uses `build_runner` to generate `freezed` data classes, BLoC states/events, and `injectable` dependency injection setup. You **must** run this step before building the app.

```bash
dart run build_runner build --delete-conflicting-outputs
```

To watch for changes during development:

```bash
dart run build_runner watch --delete-conflicting-outputs
```

### 4. Run the app

```bash
flutter run
```

## рџ”Ѓ Code Generation

This project relies on code generation for several key pieces:

| Generator | What it produces |
|---|---|
| `freezed` | Immutable data classes, `copyWith`, `==`, `hashCode`, sealed union `when`/`map` for BLoC events & states |
| `injectable_generator` | `injection.config.dart` вЂ” auto-wires all `@injectable`-annotated classes into `get_it` |

Always re-run `build_runner` after:
- Creating or modifying a `@freezed` class
- Adding or changing `@injectable` / `@lazySingleton` annotations
- Modifying repository bindings with `@Binds`
